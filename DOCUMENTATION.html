1. Vue d'ensemble du projet

MASAP-UGANC est une plateforme web de gestion académique destinée à la Faculté des Sciences Techniques et Spécialisées (FSTS) de l'UGANC. Elle permet la gestion des étudiants, enseignants, modules, notes, devoirs, documents pédagogiques et annonces.

ÉlémentDétail
FrontendNext.js 14 (App Router) — TypeScript — Tailwind CSS
BackendFastAPI (Python 3.11) — SQLAlchemy — Alembic
Base de donnéesPostgreSQL 16
AuthentificationJWT (Bearer Token)
ProxyNginx (reverse proxy)
ConteneurisationDocker + Docker Compose
HébergementContabo Storage VPS (77.237.247.61)
PaaSCoolify v4 (auto-deploy depuis GitHub)

4. Configuration locale (développement)

Prérequis

  • Docker Desktop installé
  • Node.js 20+ (optionnel si on utilise Docker)
  • Python 3.11+ (optionnel si on utilise Docker)

Lancement avec Docker Compose

cd MASAP_UGANC
docker-compose up --build

Les services seront disponibles :

  • Frontend : http://localhost:3000
  • Backend API : http://localhost:8000
  • Docs Swagger : http://localhost:8000/docs

Lancement sans Docker (backend)

cd backend
python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt
alembic upgrade head
python -m scripts.seed
uvicorn app.main:app --reload

Lancement sans Docker (frontend)

cd frontend
npm install --legacy-peer-deps
npm run dev

6. Installation de Coolify

Coolify est un PaaS self-hosted permettant de déployer plusieurs applications depuis GitHub avec auto-deploy, gestion des env vars et des domaines.

Installation

ssh root@77.237.247.61
curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash

Coolify sera accessible sur http://77.237.247.61:8000 (ou port 80 après config).

Fix critique — Désactiver IPv6 pour Docker

Problème : PostgreSQL 16/18 dans Docker retourne une erreur ParseAddr: unexpected IPv6 address au démarrage. Docker tente de lier l'adresse IPv6 qui n'est pas gérée correctement.

Solution : Désactiver IPv6 dans le daemon Docker sur le VPS :

ssh root@77.237.247.61
echo '{"ipv6": false}' > /etc/docker/daemon.json
systemctl restart docker

Relancer ensuite la ressource PostgreSQL dans Coolify.

Configuration GitHub

1
Dans Coolify : Sources → Ajouter GitHub App
2
Autoriser l'accès au dépôt MASAP_UGANC
3
Activer "Auto Deploy on Push" pour déploiement automatique à chaque git push

7. Déploiement via Coolify

Le projet est déployé avec 3 ressources distinctes dans Coolify :

7.1 PostgreSQL (Base de données)

  • Type : PostgreSQL 16 (via l'interface Coolify, pas via Dockerfile)
  • Database name : masap_uganc
  • Username : postgres
  • Récupérer l'URL interne dans : Ressource → Connection → Postgres URL (internal)
Important : Utiliser le "Postgres URL (internal)" de Coolify pour DATABASE_URL dans le backend. Le nom de la ressource (ex: masap-db) n'est pas le hostname Docker — Coolify utilise un ID interne (ex: m11323fx9sz625gfwyx1sygl).

7.2 Backend (FastAPI)

  • Type : Dockerfile (pointe vers ./backend/Dockerfile)
  • Port : 8000
  • Branche : main

7.3 Frontend (Next.js)

  • Type : Dockerfile (pointe vers ./frontend/Dockerfile)
  • Port : 3000
  • Branche : main

9. Dockerfiles de référence

frontend/Dockerfile

FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json ./
RUN npm install --legacy-peer-deps

FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
ENV NEXT_TELEMETRY_DISABLED=1
ENV NEXT_PUBLIC_API_URL=http://piqaiu91ter4h0m8njg1jff6.77.237.247.61.sslip.io
RUN npm run build

FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV production
ENV NEXT_TELEMETRY_DISABLED 1
RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public
USER nextjs
EXPOSE 3000
ENV PORT 3000
CMD ["node", "server.js"]
La ligne COPY --from=builder /app/public ./public est obligatoire : le mode standalone de Next.js n'inclut pas le dossier public/ automatiquement (le logo et autres assets statiques ne s'afficheraient pas).

backend/Dockerfile

10. Base de données — Bootstrap & Migrations

Sur une base de données vierge (nouveau déploiement), la séquence d'initialisation est :

1
create_all — Crée toutes les tables depuis les modèles SQLAlchemy
2
alembic stamp head — Marque la DB comme "à jour" dans la table alembic_version (sans réexécuter les migrations)
3
python -m scripts.seed — Insère les données initiales (admin, enseignants, étudiants, modules, spécialités…)
4
uvicorn — Lance le serveur API
Pourquoi cette séquence ? Si on exécute alembic upgrade head sur une DB vierge, la migration 001 tente d'ajouter des colonnes à des tables qui n'existent pas encore → erreur. create_all crée les tables d'abord, puis stamp head dit à Alembic qu'elles sont déjà à jour.

Pour les mises à jour futures (DB existante)

Remplacer le CMD du Dockerfile par :

alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000

Le script seed est idempotent (vérifie avant d'insérer) mais il vaut mieux ne pas le relancer à chaque démarrage sur une DB en production.

11. Problèmes rencontrés & solutions

P1 — PostgreSQL ne démarre pas (ParseAddr IPv6)

Erreur : ParseAddr("::1"): unexpected IPv6 address

Cause : Docker tente de binder IPv6, non supporté dans cette configuration.
Solution :

echo '{"ipv6": false}' > /etc/docker/daemon.json
systemctl restart docker

P2 — "Email ou mot de passe incorrect" (CORS)

Symptôme : Login échoue alors que les 41 utilisateurs sont bien en base. Les requêtes API sont bloquées par CORS.

Cause : ALLOWED_ORIGINS ne contenait pas l'URL sslip.io du frontend.
Solution : Ajouter l'URL complète du frontend dans ALLOWED_ORIGINS dans les env vars Coolify du backend.

P3 — Le logo ne s'affiche pas en production

Symptôme : /logo.png retourne 404 en production.

Cause : Next.js en mode standalone ne copie pas le dossier public/ dans le build final.
Solution : Ajouter dans le Dockerfile :

COPY --from=builder /app/public ./public

P4 — NEXT_PUBLIC_API_URL non appliqué

Symptôme : Le frontend appelle localhost:8000 au lieu du backend en production.

Cause : Les variables NEXT_PUBLIC_* sont injectées au build dans Next.js, pas au runtime. Les env vars Coolify sont runtime.
Solution : Hardcoder l'URL dans le Dockerfile :

ENV NEXT_PUBLIC_API_URL=http://piqaiu91ter4h0m8njg1jff6.77.237.247.61.sslip.io

P5 — Git push timeout (HTTP 408)

Symptôme : git push échoue avec timeout, dépôt trop lourd (98 MB).

Cause : Les dossiers .next/ et node_modules/ étaient dans l'historique Git.
Solution : Recréer l'historique avec une branche orpheline :

git checkout --orphan clean-branch
git add <fichiers sources uniquement>
git commit -m "first commit"
git branch -D main
git branch -m main
git push --force origin main

P6 — Erreur Alembic sur DB vierge

Erreur : relation "programs" does not exist

Cause : La migration 001 essaie d'ALTER TABLE sur des tables inexistantes.
Solution : Lancer Base.metadata.create_all() avant alembic stamp head (voir section 10).

P7 — TypeScript build error (Set iteration)

Erreur : Type 'Set<string>' can only be iterated with --downlevelIteration

Solution : Remplacer [...new Set(...)] par Array.from(new Set(...))

P8 — Mauvais nom de variable dans Coolify

Symptôme : Backend ne trouve pas la DB malgré la variable configurée.

Cause : La variable était nommée masap-db au lieu de DATABASE_URL.
Solution : Corriger le nom dans la vue "Developer" des env vars Coolify.

14. Checklist de déploiement

Premier déploiement (DB vierge)

  • ☐ VPS accessible via SSH
  • ☐ Coolify installé et accessible
  • ☐ IPv6 désactivé dans Docker daemon (/etc/docker/daemon.json)
  • ☐ GitHub connecté à Coolify
  • ☐ Ressource PostgreSQL 16 créée et démarrée
  • ☐ URL interne PostgreSQL copiée
  • ☐ Ressource Backend créée avec toutes les env vars
  • ☐ ALLOWED_ORIGINS contient l'URL sslip.io du frontend
  • ☐ Backend déployé et seed exécuté (vérifier les logs)
  • ☐ NEXT_PUBLIC_API_URL hardcodé dans frontend/Dockerfile
  • ☐ COPY --from=builder /app/public ./public présent dans frontend/Dockerfile
  • ☐ Ressource Frontend créée et déployée
  • ☐ Login fonctionnel sur l'URL frontend
  • ☐ Logo affiché sur la page de connexion et dans la sidebar

Mise à jour (redéploiement)

  • ☐ git push origin main
  • ☐ Coolify auto-déploie (ou déclencher manuellement)
  • ☐ Vérifier les logs de build
  • ☐ Tester l'authentification
  • ☐ Si nouvelles migrations Alembic : mettre à jour le CMD du backend Dockerfile
